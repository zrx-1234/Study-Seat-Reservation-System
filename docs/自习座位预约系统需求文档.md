# 自习座位预约系统需求文档

# 1. 文档概述

## 1.1 文档目的

本文档旨在明确自习座位预约系统的整体需求、业务流程、功能规格、非功能需求及项目实施要求，为系统设计、开发、测试、部署及维护提供标准化依据，确保系统开发符合学校实际使用场景，满足学生、管理员的核心需求，同时契合课程考评要求。

## 1.2 项目背景

以高等学校日常学习生活为背景，当前学校开放的自习室存在座位利用率偏低、资源分配不均的问题，具体表现为“一座难求”与“书包占座、人不在座”现象并存。为提升自习室座位的利用率和周转率，学校计划通过信息技术手段搭建自习座位预约系统，同时动态调整自习室管理策略，为学生提供便捷、高效的座位预约服务，为管理员提供科学的管理工具。

## 1.3 系统范围

本系统覆盖学校所有统一管理及院系专属的自习室、自习座位，面向学生（使用学生端）和管理员（使用管理端）两类用户，实现座位预约、签到、提醒、搜索、管理及智能化交互等功能，同时需在GitHub平台完成项目全流程实践，满足课程考评中项目管理、自动化、小组合作等相关要求。

## 1.4 术语定义

- **自习室**：用于学生自习的场所，包括普通教室、图书馆专用自习室及其他区域自习场地，分为学校统一管理和院系专属两类，院系专属自习室仅对本院系学生开放，其登记、注销、可用性维护由管理员统一负责。

- **自习座位**：自习室内登记在册的、具有唯一编号的可预约座位，其登记、注销、可用性维护由管理员统一负责，靠近固定插座或移动导轨插座的座位需标注特殊标记。

- **RBAC**：基于角色的访问控制，用于管理端权限分配，通过角色定义权限、给用户分配角色，实现不同用户仅能访问其被授权的功能。

- **用户故事**：描述用户使用系统的具体场景和需求，用于需求规划和项目迭代开发，需按要求编写并纳入DevCloud管理。

- **违约**：学生预约座位后，超过预约时间15分钟未签到，系统自动取消预约并记录的违规行为。

# 2. 总体需求

## 2.1 核心目标

- 提升自习室座位利用率和周转率，解决占座乱象，实现座位资源合理分配。

- 为学生提供便捷的座位预约、签到、搜索、提醒等服务，优化自习体验。

- 为管理员提供全面的自习室、座位、预约、违约及系统参数管理功能，支持管理策略优化。

- 实现系统智能化交互，通过自然语言助手简化学生操作。

- 完成GitHub平台全流程实践，满足课程考评中项目管理、自动化、小组合作等要求。

## 2.2 系统架构

- **学生端**：支持微信小程序（额外加5%分值）或Web浏览器实现，核心功能包括座位查询、预约、签到、提醒、历史预约管理、智能助手交互等。

- **管理端**：采用Web端实现，支持RBAC权限管理，核心功能包括自习室管理、座位管理、预约管理、违约管理、用户角色管理、系统参数配置等。

- **后台服务**：支撑学生端与管理端的功能实现，负责数据存储、业务逻辑处理、推送提醒、权限校验、智能化交互解析等。

- **GitHub平台**：用于项目全流程管理，包括需求规划、用户故事管理、代码托管、编译构建、自动化测试、部署、流水线搭建等。

# 3. 详细需求

## 3.1 学生端需求

### 3.1.1 基础功能需求

- **座位查询与搜索**：学生可查看所有可用自习室及座位信息，支持按自习室、可用时间、座位位置（靠窗/靠插座）等条件搜索，靠近插座的座位需显示特殊标记。

- **座位预约**：预约以整点小时为单位，单次最大预约时长可通过系统参数调整（默认4小时）；学生可选择自习室、座位及预约时间段，确认预约后生成预约记录。

- **签到功能**：学生按预约时间到达座位后，需完成签到；Web端通过录入自习教室屏幕显示的动态编码签到，微信小程序通过扫描教室动态二维码签到；动态编码/二维码每日与教室关联，实时更新。

- **预约提醒**：系统在预约时间前15分钟，通过邮件或微信服务号推送提醒；预约时间后10分钟未签到，再次推送提醒；预约时间15分钟后未签到，自动取消预约、释放座位，推送取消提醒，并记录违约一次。

- **历史预约管理**：学生可查看自身所有历史预约记录（含已完成、已取消、违约），支持对历史预约过的座位进行再次预约。

### 3.1.2 智能化需求

学生端需集成智能助手（聊天框），支持通过自然语言交互完成相关操作，核心场景如下：

- 查询实时空座：如“今天晚上还有空座吗？”，系统返回可用座位的编号、可用时间、所在教室等信息。

- 条件筛选座位：如“帮我找靠窗的座位”“找有插座的座位”，系统返回符合条件的座位信息。

- 查询个人预约：如“我今天定了哪里的座位”，系统返回当日个人预约记录（座位编号、所在教室、预约时间段）。

智能化功能可优先通过关键词匹配和规则实现，建议接入大语言模型，提升交互的自然性和准确性。

## 3.2 管理端需求

### 3.2.1 RBAC权限管理需求

- **角色维护**：管理员可创建、编辑、删除系统角色，定义不同角色的权限范围。

- **用户角色分配**：管理员可给系统用户（其他管理员）分配不同角色，实现权限的精细化管控。

- **权限适配**：管理界面需根据当前用户的角色，仅展示其被授权的功能模块，未授权功能不可见、不可操作。

### 3.2.2 核心管理功能需求

- **自习室管理**：管理员可对自习室进行登记（录入名称、位置、开放时间、管理类型（学校/院系）、所属院系等信息）、注销、可用性维护（设置可用/不可用状态）；可查看所有自习室的详细信息及预约情况。

- **座位管理**：管理员可对自习室内的座位进行登记（录入座位编号、位置特征（靠窗/靠插座）、所属自习室等信息）、注销、可用性维护；可批量管理座位，查看座位的预约及使用记录。

- **预约管理**：管理员可查看所有学生的预约记录（含预约人、座位、时间段、状态），支持为学生手动预约、取消预约，处理预约相关异常问题。

- **违约管理**：管理员可查看所有学生的违约记录（含违约人、违约时间、违约座位、违约原因），支持统计违约次数，制定相关管理策略。

- **系统参数配置**：管理员可调整系统核心参数，包括单次最大预约时长、预约提醒时间、违约判定时间、动态编码/二维码更新频率等。

- **数据统计与分析**：管理员可查看自习室、座位的预约率、利用率等数据，通过数据优化管理策略，提升资源利用率。

## 3.3 业务流程需求

### 3.3.1 座位预约与签到流程

1. 学生登录学生端，查询可用座位（可按条件搜索）。

2. 选择目标座位及预约时间段（整点单位，不超过最大预约时长），确认预约。

3. 系统生成预约记录，在预约前15分钟推送提醒。

4. 学生按预约时间到达教室，通过Web端录入动态编码或微信端扫描二维码完成签到。

5. 若预约后10分钟未签到，系统再次推送提醒；15分钟未签到，自动取消预约、释放座位，记录违约并推送提醒。

6. 学生完成自习后，无需手动退座，预约时间结束后系统自动释放座位；若提前离开，可手动取消预约（需符合系统规则）。

### 3.3.2 管理员管理流程

1. 管理员登录管理端，根据自身角色权限访问对应功能模块。

2. 对自习室、座位进行登记、维护，配置系统参数。

3. 查看预约、违约记录，处理异常预约（如手动取消违规占座预约）。

4. 统计分析座位利用率，调整自习室开放策略或座位配置。

5. 维护系统角色及用户权限，确保管理操作的安全性和规范性。

## 3.4 非功能需求

- **可用性**：系统运行稳定，学生端、管理端响应时间≤3秒；动态编码/二维码更新及时，签到成功率≥99%；提醒推送及时，延迟≤1分钟。

- **安全性**：用户登录需进行身份验证（学生用学号/密码，管理员用账号/密码）；学生仅能查看、操作自身预约记录，管理员权限严格按角色分配，防止越权操作；数据（预约记录、违约记录、用户信息）加密存储，防止泄露。

- **可扩展性**：系统支持后续功能扩展，如增加通宵自习室管理、座位预约排队、违约惩罚机制等；支持自习室、座位数量的批量新增，适配学校规模扩大需求。

- **易用性**：学生端界面简洁，操作流程简单，无需复杂培训即可使用；管理端界面清晰，功能模块分类明确，便于管理员高效操作。

- **兼容性**：学生端Web版支持主流浏览器（Chrome、Edge、Firefox等），微信小程序支持最新版本微信；管理端支持主流浏览器，适配不同屏幕尺寸。

# 4. GitHub实践需求

## 4.1 实践目标

在GitHub平台完成项目完整流程，包括需求规划、用户故事管理、代码托管、编译构建、自动化测试、部署、流水线搭建等，满足课程考评中项目管理、自动化、小组合作的要求。

## 4.2 具体实践步骤

1. **账号注册**：团队成员自行注册GitHub账号。

2. **项目创建与团队管理**：组长创建Scrum项目，将团队所有成员加入项目，明确成员职责。

3. **需求规划**：在项目工作区中，通过思维导图编写需求规划，明确需求规划中4个栏目的含义，梳理系统所有需求。

4. **用户故事与测试用例管理**：编写所有用户故事（先用Markdown编写，Review后纳入GitHub Projects管理）；关联测试用例，包括功能手工测试和接口自动测试，探索用户故事完成度的判断方法。

5. **代码托管**：创建代码库，将项目代码托管至GitHub平台，规范代码提交、分支管理流程。

6. **编译构建**：创建编译构建任务（使用GitHub Actions），集成自动化单元测试，确保代码编译通过且核心功能可正常运行。

7. **部署任务**：创建部署任务，将编译后的项目部署至对应环境（开发环境、测试环境、生产环境）。

8. **流水线搭建**：创建流水线（使用GitHub Actions），整合编译构建、自动化测试、部署等环节，实现项目开发、测试、部署的自动化流程。

# 5. 项目实施与考评要求

## 5.1 项目阶段划分

### 5.1.1 第一阶段（第5周开始Review）

- 识别尽可能多的用户故事，用Markdown编写，Review后纳入GitHub Projects管理。

- 完成1-2个迭代功能，搭建系统初步架构，实现核心用户故事（如基础的座位查询、预约功能）。

- 完成代码仓库搭建、初步的编译构建任务（GitHub Actions），规范小组合作流程（代码提交、任务分配等）。

### 5.1.2 第二阶段（第12周开始Review）

- 识别并定义所有用户故事，完善需求规划。

- 实现超过一半的系统功能，优先完成除自然语言交互外的基础核心功能。

- 在GitHub上完成DevOps完整流程（代码托管、编译构建、自动化测试、部署、流水线）。

- 完善小组合作管理，解决代码合并冲突，优化代码质量。

### 5.1.3 项目汇报展示（15、16周）

展示系统完整功能，根据课程进度，可选择提交视频、项目代码及相关文档（需求文档、设计文档、测试文档等）。

## 5.2 考评标准

- **小组考评（70%）**：
        

    - 两次Review（40%）：涵盖Teamwork（小组合作）、需求梳理、系统设计、开发进度、测试情况、DevOps流水线及代码仓库管理，通过两次Review完成考评。

    - 项目汇报展示（30%）：根据系统功能实现情况、项目完成质量、汇报效果（或提交的视频、代码、文档）进行考评。

- **个人考评（30%）**：以课程论文形式完成，内容可围绕系统开发、功能实现、个人负责模块、项目实践心得等展开。
