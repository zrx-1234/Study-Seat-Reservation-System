# 自习座位预约系统小组分工安排文档

## 1. 引言

### 1.1 小组成员
小组共有6名成员，按照功能模块分为三个部分，每部分2人：
- **系统-管理员部分**：成员A、成员B
- **系统-学生部分**：成员C、成员D
- **AI智能化部分**：成员E、成员F

### 1.2 分工原则
- 每个部分负责独立的功能模块，但共享后端API和数据层。
- 采用Scrum方法，使用GitHub进行项目管理、代码托管和CI/CD。
- 定期Review，确保进度和质量。

## 2. 各部分职责

### 2.1 系统-管理员部分（成员A、成员B）
**负责工作**：
- 开发管理端Web应用（基于Vue.js）。
- 实现RBAC（基于角色的访问控制）：角色维护、用户角色分配、权限控制。
- 自习室管理：登记、注销、可用性维护（包括开放时间、院系限制）。
- 座位管理：登记、注销、可用性维护（包括位置、特殊标记如插座、靠窗）。
- 预约记录管理：查看预约、违约记录，为用户代预约/取消。
- 系统参数调整：最大预约时间、最大预约小时数等。
- 界面设计：根据角色仅展示授权功能。

**技术栈**：前端框架（Vue）、后端API集成、数据库交互。

### 2.2 系统-学生部分（成员C、成员D）
**负责工作**：
- 开发学生端应用（Web应用或微信小程序）。
- 座位查看：显示可用自习室、座位信息（编号、时间、教室）。
- 预约功能：选择时间（整点小时，最多4小时）、提交预约。
- 签到功能：扫描二维码或输入编码验证签到。
- 搜索功能：按条件搜索座位（靠窗、插座等）、历史预约再次预定。
- 提醒接收：预约前15分钟提醒、签到后10分钟提醒、违约通知。
- 用户界面：响应式设计，支持移动端。

**技术栈**：前端框架（Vue或微信小程序）、后端API调用、推送集成。

### 2.3 AI智能化部分（成员E、成员F）
**负责工作**：
- 开发智能助手聊天框（集成到学生端）。
- 自然语言处理：基于关键词匹配或大语言模型，实现查询解析。
- 支持查询：可用座位、靠窗座位、历史预约等（Q1-Q3）。
- 后端集成：调用预约服务、座位搜索API。
- 对话管理：维护会话状态，提供上下文相关回复。
- 扩展功能：接入大语言模型API，提升交互自然度。

**技术栈**：前端聊天组件、NLP库（或API调用）、后端服务集成。

## 3. 不同部分之间的交互

### 3.1 API接口共享
- **后端服务**：所有部分共享统一的API Gateway，后端服务包括预约服务、签到服务、通知服务、管理服务、智能助手服务。
- **数据层**：共享数据库（MySQL），确保数据一致性。
- **交互方式**：
  - 系统-学生部分调用预约服务API进行预约/签到。
  - 系统-管理员部分调用管理服务API进行CRUD操作。
  - AI智能化部分调用智能助手服务API，并间接调用其他服务（如搜索座位）。

### 3.2 协作机制
- **代码仓库**：使用GitHub代码库，分支管理（feature分支合并到main）。
- **API文档**：维护Swagger/OpenAPI文档，确保接口一致。
- **会议**：每周站会，讨论进度、接口变更、集成问题。
- **测试**：共享单元测试、集成测试，确保模块间兼容。

### 3.3 数据交互
- **用户数据**：学生和管理端共享用户表。
- **座位/预约数据**：学生端读写预约，管理员端管理座位。
- **通知**：通知服务推送给学生端用户。

## 4. 完成整个项目的工作流

### 4.1 总体工作流（基于Scrum和GitHub）
1. **需求分析阶段**（第1周）：
   - 识别用户故事（所有成员参与）。
   - 定义需求规格，分配到各部分。

2. **设计阶段**（第2-3周）：
   - 架构设计（共享文档）。
   - API设计、数据库设计。
   - 各部分UI/UX设计。

3. **开发阶段**（第4-10周）：
   - 并行开发：各部分独立开发前端/后端模块。
   - 每日提交代码，自动化测试。
   - 每周集成：合并分支，解决冲突。

4. **测试阶段**（第11-12周）：
   - 单元测试、集成测试、端到端测试。
   - 用户验收测试（UAT）。

5. **部署阶段**（第13-14周）：
   - GitHub Actions流水线：构建、测试、部署。
   - 生产环境上线。

6. **维护阶段**（第15-16周）：
   - Bug修复、性能优化。
   - 项目汇报准备。

### 4.2 具体工作流步骤
- **迭代周期**：2周一个Sprint。
- **任务分配**：在GitHub Projects中创建用户故事，分配给各部分成员。
- **进度跟踪**：使用思维导图和Scrum板。
- **质量保证**：代码审查、自动化测试覆盖率>80%。
- **交付物**：代码、测试报告、文档、演示视频。

### 4.3 时间安排（基于lab阶段）
- **第一阶段（第5周Review）**：完成初步架构和1-2个用户故事。
- **第二阶段（第12周Review）**：实现>50%功能，完整DevOps流程。
- **项目汇报（15-16周）**：完整功能展示。

## 5. 风险管理
- **进度风险**：定期Review，调整分工。
- **技术风险**：培训新技术，共享知识。
- **沟通风险**：使用GitHub协作工具。

## 6. 结论
此分工安排确保各部分独立开发，同时通过共享接口实现集成。遵循DevOps实践，保证项目高质量完成。
